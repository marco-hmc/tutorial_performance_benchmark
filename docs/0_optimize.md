## 性能优化

### 1. concepts

#### 1.1 这些是一些常见操作的成本的数量级的概念:

时钟频率表示 CPU 每秒钟可以执行的周期数。时钟频率越高，CPU 每秒钟可以执行的周期数就越多，处理速度也就越快。现代家用 CPU 的时钟频率通常在几 GHz 范围内。例如，3 GHz 的 CPU 表示每秒钟有 30 亿个周期。

**基础计算：**

```
CPU 频率：3GHz = 3 × 10^9 Hz
时钟周期时间：T = 1/f = 1/(3×10^9) ≈ 0.33ns

时间单位换算：
1秒 = 10^3毫秒 = 10^6微秒 = 10^9纳秒
```

**1. 内存层次的巨大差异：**

```
性能差距：
L1缓存 vs 主内存：100倍差距
主内存 vs SSD：1000倍差距
SSD vs HDD：100倍差距
本地 vs 广域网络：1000倍差距
```

> - **L1 缓存**：一般 L1 指令缓存的大小在 32KB 到 128KB 之间，L1 数据缓存的大小也在 32KB 到 128KB 左右。
> - **L2 缓存**：大小一般在 256KB 到 2MB 之间。
> - **L3 缓存**：一般在 4MB 到 32MB 之间。

**2. 并发操作的隐性成本：**

- **线程创建**：比简单计算慢 10 万倍
- **上下文切换**：包含缓存失效的真实成本可达毫秒级
- **系统调用**：即使简单的系统调用也比内存访问慢 1000 倍

- **汇编指令**：汇编指令是人类可读的指令，通常直接对应于机器指令（CPU 指令）。每条汇编指令通常对应一条或多条机器指令。
- **指令执行过程**

  1. **取值（Fetch）**：从内存中取出指令。
  2. **译码（Decode）**：将取出的指令译码为 CP U 可以理解的操作。
  3. **执行（Execute）**：执行指令的操作，例如算术运算、逻辑运算、内存访问等。
  4. **写回（Write Back）**：将执行结果写回寄存器或内存。

- **时钟周期消耗**

  - **取值、译码、写回**：这些阶段的时钟周期消耗相对固定，通常都是一个时钟周期。
  - **执行**：执行阶段的时钟周期消耗取决于指令的类型，可以是一个时钟周期到几十个时钟周期。例如，简单的算术运算可能只需要一个时钟周期，而复杂的指令（如除法、内存访问）可能需要多个时钟周期。

- **超标量结构**
  - **超标量结构**：现代 CPU 通常使用超标量结构，可以在一个核内一个时钟周期内执行多条指令。这种结构通过增加多个执行单元，使得 CPU 可以并行处理多条指令，从而提高指令执行的效率。

#### 1.2 优化原则 && 代码优化主要思路

**1. 二八原则：** 在软件开发中，二八原则具有重要的指导意义。大量实践表明，80% 的性能问题往往集中出现在 20% 的代码部分。这意味着在进行性能优化时，不能盲目地对整个代码库进行全面调整，而应该通过性能分析工具，精准定位到这关键的 20% 的代码，将优化工作重点放在这些部分，从而更有效地提升整体性能。例如，在一个复杂的图形渲染程序中，可能大部分的时间都花费在少量核心的渲染算法模块上，找到并优化这些模块就能显著提高程序的运行效率。

**2. 先正确运行，再优化：** 确保程序能够正确无误地运行是进行性能优化的前提条件。如果程序本身存在功能错误，即使进行了性能优化，也无法满足实际需求。先专注于实现程序的正确逻辑，保证其在各种预期情况下都能给出准确结果，然后再去考虑如何提高其运行效率。比如，在开发一个财务计算软件时，首先要保证计算结果的准确性，否则即便程序运行速度再快，错误的结果也毫无价值。

**3. 不要太提前优化：** 过早进行优化是软件开发中的一个常见误区。在项目初期，代码的架构和需求可能还不稳定，此时花费大量时间进行性能优化，不仅可能会增加代码的复杂性，降低代码的可维护性，还可能因为需求的变更而导致之前的优化工作白费。应该先确保代码具有良好的结构和可读性，便于后续的开发和维护。只有在程序的功能基本稳定，性能问题开始凸显时，再进行针对性的优化。例如，在一个快速迭代的互联网应用开发中，初期更重要的是快速实现功能并推向市场，过早追求极致性能可能会阻碍产品的快速上线。

**4. 优化需要数据支持：** 在进行性能优化时，不能仅凭经验或直觉来判断哪些优化措施是有效的。现代编译器具有很强的优化能力，一些看似能够提升性能的优化方法，实际上编译器可能已经在编译过程中进行了类似的优化，或者由于硬件特性等原因，这些方法并不能真正带来性能提升，甚至可能导致性能下降。因此，必须依靠性能分析工具提供的数据，如代码各部分的执行时间、资源占用情况等，来确定真正需要优化的点以及选择合适的优化策略。例如，在某些情况下，手动展开循环被认为可以提高性能，但现代编译器已经能够自动进行循环展开优化，手动展开可能并不会带来额外的性能收益。

#### 1.3 CPU 密集任务

1. **多线程**
   - **利用多核处理器**：随着硬件技术的发展，多核处理器已经成为主流。对于 CPU 密集型任务，使用多线程可以将计算任务并行化，充分利用多核处理器的计算能力，从而显著提高任务的执行效率。例如，在进行大规模数据的数值计算时，可以将数据分成多个部分，每个线程负责处理一部分数据，最后将结果合并。
   - **线程同步协调**：在多线程编程中，线程之间的同步是至关重要的。使用 `wait` 机制（如条件变量 `std::condition_variable` 等）可以有效地协调线程间的同步。例如，当一个线程需要等待另一个线程完成某个任务后才能继续执行时，可以使用条件变量来实现这种等待和唤醒机制，确保线程之间能够正确协作，避免出现数据竞争和死锁等问题。
2. **优化算法实现**
   - **避免不必要的递归**：递归虽然在某些情况下可以使代码更简洁，但由于其频繁的函数调用和栈空间开销，在性能上往往不如迭代。将递归算法转换为迭代算法，可以减少函数调用的开销，提高执行效率。例如，计算阶乘的递归算法可以很容易地转换为迭代算法。
   - **使用引用传递参数**：在函数调用时，如果参数是通过值传递的方式传递，会发生参数的复制，这在处理大型对象时会带来较大的性能开销。使用引用传递参数可以避免这种不必要的复制，提高函数调用的效率。例如，对于一个包含大量数据成员的自定义类对象，通过引用传递可以避免对象的深拷贝，节省时间和空间。
   - **减少重复计算**：仔细分析代码逻辑，找出那些不必要的重复计算，并通过适当的缓存机制或优化算法来避免。例如，在循环中每次都重新计算相同的表达式，可以将该表达式的计算结果缓存起来，在后续循环中直接使用，减少计算量。
   - **选择高效的算法和数据结构**：不同的算法和数据结构在不同的场景下具有不同的性能表现。根据具体的应用场景，选择最合适的算法和数据结构可以显著提升性能。例如，在需要频繁查找元素的场景下，使用哈希表或平衡二叉搜索树（如 `std::unordered_map` 或 `std::map`）比使用链表或数组具有更高的查找效率。

#### 1.4 I/O 密集任务

1. **内存 I/O 密集**
   - **对象池的应用**：在频繁创建和销毁对象的场景下，对象池可以显著提高性能。对象池预先创建并管理一定数量的对象，当需要使用对象时，从对象池中获取，使用完毕后再归还到对象池，避免了频繁的对象构造和销毁操作所带来的开销。例如，在一个网络服务器程序中，可能会频繁地创建和销毁网络连接对象，使用对象池可以减少这些开销，提高服务器的性能。
   - **优化内存访问模式**：现代计算机的缓存机制对内存访问模式非常敏感。使用顺序访问内存可以提高缓存命中率，从而提高内存访问效率。例如，在处理大型数组时，按行顺序访问数组元素比按列顺序访问更有利于缓存命中，因为现代计算机的缓存通常是按行缓存数据的。
2. **硬盘 I/O 密集**
   - **异步 I/O 操作**：硬盘 I/O 操作通常比内存操作和 CPU 计算要慢得多。使用异步 I/O 操作，程序可以在发起 I/O 请求后，不必等待 I/O 操作完成，而是继续执行其他任务，从而提高程序的并发性能。例如，在读取大型文件时，使用异步 I/O 可以让程序在等待文件读取的过程中处理其他逻辑，提高整体效率。
   - **缓存的使用**：通过使用缓存可以减少对磁盘的 I/O 操作次数。将经常访问的数据缓存到内存中，当再次需要访问这些数据时，首先从缓存中查找，如果存在则直接使用，避免了从磁盘读取数据的开销。例如，在数据库应用中，使用缓存可以显著提高查询性能。
   - **选择高效的文件系统和磁盘调度算法**：不同的文件系统和磁盘调度算法对 I/O 性能有很大影响。在选择文件系统时，要根据应用场景选择适合的文件系统，如在需要频繁读写小文件的场景下，某些文件系统可能具有更好的性能。同时，合理配置磁盘调度算法也可以优化 I/O 性能，例如，对于随机 I/O 较多的应用，选择适合随机 I/O 的调度算法可以提高响应速度。
3. **网络 I/O 密集**
   - **异步网络 I/O 操作**：与硬盘 I/O 类似，网络 I/O 也存在较大的延迟。使用异步网络 I/O 操作可以提高程序的并发性和效率。通过异步 I/O，程序可以在等待网络数据传输的过程中执行其他任务，避免线程阻塞，从而提高整体性能。例如，在开发网络服务器时，使用异步网络 I/O 可以同时处理多个客户端的请求，提高服务器的并发处理能力。
   - **选择高效的网络协议**：不同的网络协议在网络延迟、带宽利用率等方面存在差异。根据应用的需求选择合适的网络协议可以减少网络延迟，提高数据传输效率。例如，对于实时性要求较高的应用，如视频流传输，选择 UDP 协议可能比 TCP 协议更合适，因为 UDP 协议具有更低的延迟；而对于数据准确性要求极高的应用，如文件传输，TCP 协议则更为可靠。
   - **数据压缩**：在网络传输中，数据量的大小直接影响传输效率。通过使用压缩算法对传输的数据进行压缩，可以显著减少传输的数据量，从而提高传输效率。例如，在 Web 应用中，服务器可以对返回给客户端的数据进行压缩，减少网络带宽的占用，加快页面加载速度。

在进行代码优化时，必须认识到优化方法在没有数据支持的情况下是不可靠的。随着计算机技术的不断发展，编译器和硬件都在不断优化和改进，一些传统的优化方法可能已经被编译器或硬件底层默认支持，或者由于新的技术架构和硬件特性而不再适用。因此，在进行优化之前，一定要依靠性能分析工具提供的数据，科学地选择优化方法，避免盲目优化导致性能不升反降。

#### 1.5 硬件友好代码

- **硬件友好代码**
  1. **利用 SIMD 特性**：
  - 现在的 CPU 很多时候都可以对指令并行化，利用 SIMD（Single Instruction, Multiple Data）特性进行向量化计算，提高并行计算性能。
  2. **提升缓存利用率**：

### 2. 实现优化

#### 2.1 RVO

- **什么是 RVO？RVO 有什么好处？** RVO，全称 Return Value Optimization（返回值优化），是 C++编译器的一种优化技术。当函数返回一个局部对象时，RVO 允许编译器省略额外的拷贝或移动构造函数的调用，直接在调用位置构造返回对象。

RVO 的主要好处是提高性能。通过省略额外的拷贝或移动构造函数的调用，RVO 可以减少不必要的临时对象的创建，从而减少内存使用和提高运行速度。对于大型对象或者拷贝操作开销大的对象，RVO 的性能优势更为明显。

#### 2.2 soa 和 aos

- **soa 和 aos** 让我们通过一个具体的例子来解释 SOA（Structure of Arrays）和 AOS（Array of Structures）。 假设我们有一个粒子系统，每个粒子都有位置（x，y，z）和速度（vx，vy，vz）。

  ```cpp
  namespace AOS {
      struct Particle
      {
          float x, y, z;
          float vx, vy, vz;
      };
      std::array<Particle, 1000> particles;
      // 如果我们想更新所有粒子的位置，我们需要遍历整个数组
      // 这可能会导致缓存未命中，因为位置和速度数据在内存中是交错的。
      update(particles);
  }

  namespace SOA {
      struct Particles
      {
          std::array<float, 1000> x, y, z;
          std::array<float, 1000> vx, vy, vz;
      };

      Particles particles;
      // 如果我们想更新所有粒子的位置，我们可以连续地访问 x、y 和 z 数组，这有助于提高缓存命中率
      // 因为这些数据在内存中是连续的。
      update(particles);
  }
  ```

#### 2.3 CRTP

CRTP，全称 Curiously Recurring Template Pattern（奇异递归模板模式），是一种在 C++ 中使用的编程技巧。这种模式涉及到一个类模板，它以自己的派生类作为模板参数。**CRTP 可以用于实现编译时的多态性**，也就是说，**它可以在编译时决定调用哪个函数，而不是在运行时**。这可以提高性能，因为它**避免了虚函数调用的开销**。

    ```cpp
    template <typename Derived>
    class Base {
    public:
        void interface() {
            static_cast<Derived*>(this)->implementation();
        }
    };

    class Derived : public Base<Derived> {
    public:
        void implementation() {
            // 实现具体的功能
        }
    };
    ```

在这个例子中，`Base` 是一个模板类，它有一个 `interface` 方法，这个方法调用 `implementation` 方法。`Derived` 类继承自 `Base`，并提供 `implementation` 方法的实现。

当我们调用 `interface` 方法时，实际上调用的是 `Derived` 类的 `implementation` 方法。这是在编译时决定的，所以没有运行时的虚函数调用开销。

#### 2.4 CAS（Compare-and-Swap）

CAS，全称 Compare-and-Swap（比较并交换），是一种用于实现并发算法的原子操作。这种操作可以在多线程环境中安全地读取和更新共享数据，而无需使用锁。

CAS 操作接受三个参数：一个内存位置、一个预期的旧值和一个新值。如果内存位置的当前值与预期的旧值匹配，那么 CAS 操作就会将新值写入内存位置。否则，操作不会进行任何更改。无论哪种情况，CAS 操作都会返回内存位置的原始值。CAS 操作的一个关键特性是它是原子的，也就是说，它不会被其他线程的操作中断。这使得它在实现无锁数据结构和其他并发算法时非常有用。

    ```cpp
    int compare_and_swap(int* ptr, int old_val, int new_val) {
        int original_val = *ptr;
        if (original_val == old_val) {
            *ptr = new_val;
        }
        return original_val;
    }
    ```

在实际使用中，CAS 操作通常由硬件直接支持，并通过特殊的机器指令实现。在 C++ 中，你可以使用 `<atomic>` 头文件中的 `std::atomic` 类来进行 CAS 操作。

#### 2.5 COW（Copy On Write）

- **什么是 COW（Copy On Write）？有什么用？** COW，即 Copy On Write，是一种优化策略。在这种策略下，当对象被复制时，并不立即进行实际的复制操作，而是等到对象被修改时才进行复制。这种策略可以避免不必要的复制操作，从而提高程序的性能。

- **在什么场景下使用？**

  1.  **需要复制大量数据但不经常修改的场景**：

      - COW 技术通常在需要复制大量数据，但又不经常修改数据的场景中使用。例如，在字符串操作中，如果一个字符串被复制多次，但只有少数几次会修改字符串，那么使用 COW 技术可以大大提高性能。

  2.  **共享资源的场景**：
      - 在多线程或多进程环境中，多个线程或进程可以共享同一个资源，直到其中一个需要修改资源时才进行复制。这可以显著减少内存使用和提高性能。

- **怎么实现？** COW 技术的实现通常依赖于引用计数。以下是实现 COW 的步骤：
  1.  **引用计数**：
      - 每个对象都有一个引用计数，表示有多少个引用指向这个对象。当对象被复制时，不复制对象本身，而是增加引用计数。
  2.  **延迟复制**：
      - 当对象被修改时，首先检查引用计数。如果引用计数大于 1，那么就复制对象，然后修改复制后的对象，同时减少原对象的引用计数。如果引用计数等于 1，那么直接修改对象。

COW（Copy On Write）是一种优化策略，通过延迟复制操作直到对象被修改时才进行实际的复制，从而避免不必要的复制操作，提高程序性能。COW 技术通常在需要复制大量数据但不经常修改的场景中使用，其实现通常依赖于引用计数。通过合理使用 COW，可以显著减少内存使用和提高性能。

#### 2.6 考虑使用延迟计算

- **什么是延迟计算？** 一个延迟计算的例子：

  class String{....} String s1 = "Hello"; String s2 = s1; //在正常的情况下，这一句需要调用 new 操作符分配堆内存，然后调用 strcpy 将 s1 内的数据拷贝到 s2 里面。但是我们此时 s2 并没有被使用，所以我们不需要 s2，这个时候如果让 s2 和 s1 共享一个值，就可以减小这些开销

  使用延迟计算进行读操作和写操作：

  String s = "Homer's Iliad"; cout << s[3]; s[3] = 'x';

  首先调用 operator[] 用来读取 string 的部分值，但是第二次调用该函数式为了完成写操作。读取效率较高，写入因为需要拷贝，所以效率较低，这个时候可以推迟作出是读操作还是写操作的决定。

  延迟策略进行数据库操作：有点类似之前写 web 的时候，把数据放在内存和数据库两份，更新的时候只更新内存，然后隔一段时间（或者等到使用的时候）去更新数据库。 在 effective c++里面，则是更加专业的将这个操作封装成了一个类，然后把是否更新数据库弄成一个 flag。以及使用了 mutable 关键字，来修改数据

  延迟表达式：

  Matrix<int> m1(1000, 1000), m2(1000, 1000); m3 = m1 + m2; 因为矩阵的加法计算量太大（1000\*1000）次计算，所以可以先用表达式表示 m3 是 m1 和 m2 的和，然后真正需要计算出值的时候再真的进行计算（甚至计算的时候也只计算 m3[3][2]这样某一个位置的值）

#### 4.2 pgo

传统编译器优化基于静态分析，无法知道程序的实际运行模式。PGO 通过收集真实运行数据，让编译器做出更精准的优化决策。

**PGO 的核心价值：**

```
静态优化 vs PGO优化：
❌ 编译器猜测：哪个分支更可能执行？
✅ PGO知道：分支A执行95%，分支B执行5%

❌ 编译器猜测：循环迭代次数？
✅ PGO知道：平均迭代1000次，适合展开
```

PGO，全称为 Profile-Guided Optimization(基于分析的优化)，是一种编译器优化技术，它通过收集程序运行时的性能信息来指导编译器优化。

PGO 的工作流程通常包括以下几个步骤:

1. **编译**:首先，你需要使用特殊的编译器选项来编译你的程序，这会使得程序在运行时收集性能信息。
2. **运行**:然后，你需要运行你的程序，并让它处理一些典型的工作负载。这会生成一个性能分析文件，其中包含了程序运行时的性能信息。
3. **优化编译**:最后，你需要再次使用特殊的编译器选项来编译你的程序，这次编译器会使用性能分析文件中的信息来指导优化。

通过这种方式，编译器可以了解到程序的实际运行情况，例如哪些代码经常被执行，哪些代码很少被执行，哪些条件分支的预测结果通常是什么等等。然后，编译器可以使用这些信息来进行更有效的优化，例如更精确地进行内联，更智能地安排代码布局，更准确地预测条件分支等等。

PGO 可以显著提高程序的运行速度，特别是对于那些性能敏感的应用程序。然而，PGO 也有一些缺点，例如它会增加编译时间和编译复杂性，因此并不是所有的项目都适合使用 PGO。

#### 4.4 尾递归优化

编译器对特定模式的递归，能转成迭代：

**编译器尾递归优化：**

```cpp
// 原始尾递归函数
int sum_tail_recursive(int n, int acc = 0) {
    if (n == 0) return acc;
    return sum_tail_recursive(n - 1, acc + n);  // 尾调用
}
// 传统递归的问题：
// 1. 每次调用需要新的栈帧（栈空间消耗）
// 2. 函数调用开销（参数传递、返回地址保存）
// 3. 深度递归可能导致栈溢出
// 4. 缓存不友好（函数调用破坏局部性）

// 编译器优化后的等效代码
int sum_optimized(int n, int acc = 0) {
    while (n != 0) {  // 递归转换为循环
        acc += n;     // 累积计算
        n -= 1;       // 递减计数
    }
    return acc;       // 返回结果
}
```

**尾递归优化的条件：**

```cpp
// ✅ 可以优化的尾递归
int good_tail_recursion(int n, int result) {
    if (n == 0) return result;
    return good_tail_recursion(n - 1, result + n);  // 直接返回递归调用
}

// ❌ 无法优化的递归（不是尾递归）
int bad_recursion(int n) {
    if (n <= 1) return 1;
    return n * bad_recursion(n - 1);  // 递归调用后还有乘法操作
}

// ❌ 无法优化的伪尾递归
int pseudo_tail_recursion(int n, int acc) {
    if (n == 0) return acc;
    int temp = pseudo_tail_recursion(n - 1, acc + n);
    return temp;  // 虽然看起来是尾调用，但有额外操作
}
```

### 99. quiz

#### 1. 虚函数表开销

虚函数表（vtable）的使用确实会引入一些性能开销，主要体现在以下几个方面：

- **1. 虚函数调用的间接寻址**

  - **解释**：虚函数调用需要通过虚函数表进行间接寻址，这比直接调用函数要慢。
  - **原因**：编译器在编译时无法确定具体调用哪个函数，因此在运行时需要通过虚函数表查找函数地址，然后进行调用。这种间接寻址会增加指令开销。

- **2. 破坏指令缓存（Instruction Cache）**

  - **解释**：虚函数调用可能会破坏指令缓存（Instruction Cache），因为虚函数表中的函数地址可能指向不同的内存位置。
  - **原因**：现代处理器使用指令缓存来加速指令的执行。当程序执行过程中频繁跳转到不同的内存位置时，指令缓存可能会失效（cache miss），导致性能下降。

- **3. 破坏数据缓存（Data Cache）**
  - **解释**：虚函数表的访问可能会破坏数据缓存（Data Cache），因为虚函数表和实际函数实现可能位于不同的内存区域。
  - **原因**：数据缓存用于加速数据的访问。当程序访问虚函数表时，可能会导致数据缓存失效（cache miss），从而增加内存访问延迟。

#### 2. 什么时候该使用对象池？

对于平凡构造的对象来说，使用对象池没有多大意义。因为如果复用对象的数据需要修改，其实和重新构造一个差不多了。如一般的`Point`对象，使用池化，就是多此一举的。因此对象池一般是针对构造函数实现有特殊处理的，大多为涉及资源，而非数据的。比如说`socketFd`，这是一个系统资源，要走到内核态去申请和释放，这个使用池化技术是有意义的。
